# Day 2 — Create Shared Compatibility Contracts

## Purpose

Define the shared contracts that Diagram 1 and Diagram 2 will both use.

Do this before building Diagram 2 so compatibility is designed in rather than retrofitted.

## Scope

Create or formalize shared modules for:

```text
Canonical Diagram state
Object identity and ID remapping
Object Template schema
Selection clipboard package
PMT Diagram file codec
Diagram document access
Compatibility capability checks
```

Avoid changing Diagram 1 rendering today.

## 1. Canonical editable state

The canonical state must remain renderer-neutral.

Both Diagram 1 and Diagram 2 must consume the same normalized state shape. Do not create:

```text
Diagram1State
Diagram2State
```

Prefer one shared normalization API, using the existing annotation/Diagram state as the baseline.

If Diagram 2 needs renderer caches, versions, mounted-node maps, or dirty flags, keep those outside persisted canonical state.

## 2. Shared file codec

Keep the existing format:

```json
{
  "format": "pmt-diagram",
  "formatVersion": 1
}
```

Do not introduce `pmt-diagram-2`.

Continue using the existing shared functions in:

```text
wwwroot/js/features/diagram/pmt-diagram-file.js
```

Improve the codec only when needed for compatibility.

Required rules:

- `generator.feature` is advisory and must not block reading.
- Diagram 1 and Diagram 2 must both parse files generated by either screen.
- Unknown top-level and Diagram `extensions` should be preserved where practical.
- Do not persist Diagram 2-only renderer caches.
- If a future field is not understood by Diagram 1, Diagram 1 must either preserve it safely or Diagram 2 must not write it.
- Full SVG remains an export artifact; editable state remains authoritative.

Add a compatibility matrix test:

```text
Diagram 1 export -> Diagram 1 import
Diagram 1 export -> Diagram 2 import
Diagram 2 export -> Diagram 1 import
Diagram 2 export -> Diagram 2 import
```

Diagram 2 may not exist yet; provide test seams or deferred tests without faking passing behavior.

## 3. Shared clipboard package

Define one versioned renderer-neutral clipboard package:

```json
{
  "format": "pmt-diagram-selection",
  "formatVersion": 1,
  "minimumReaderVersion": 1,
  "source": {
    "application": "PMT",
    "feature": "Diagram"
  },
  "selection": {
    "objects": [],
    "relationships": [],
    "manualRelationshipRoutes": {},
    "extensions": {}
  },
  "extensions": {}
}
```

Adjust the precise shape to match PMT's canonical model, but retain:

- Explicit format.
- Explicit version.
- Selected objects.
- Relationship information required to preserve relationships between copied Entities.
- Manual route information where valid.
- Extension containers.
- No live DOM, renderer cache, dirty state, selection handles, or screen coordinates.

Clipboard transport:

1. Prefer a structured Clipboard API representation when supported.
2. Always include a reliable `text/plain` fallback using a distinctive header such as:

```text
PMT_DIAGRAM_SELECTION_V1
{JSON}
```

3. Keep an in-memory clipboard only as an optimization, never as the only cross-screen path.
4. Pasting after navigating from Diagram 1 to Diagram 2 in the same tab must work.
5. Pasting into another PMT browser tab should work when browser permissions permit.

## 4. ID remapping rules

On paste:

- Generate new object IDs.
- Remap group IDs.
- Remap annotation-to-Entity references.
- Remap relationship endpoint references.
- Remap manual relationship route IDs.
- Preserve relationships only when their required copied endpoints are present.
- Preserve relative positions.
- Offset repeated paste instances.
- Never overwrite an existing object with a clipboard ID collision.

Expose shared pure helpers and unit-test them.

## 5. Shared Object Template contract

Both screens must use the existing API:

```text
GET /api/image-annotation/template-library
GET /api/image-annotation/default-template-library
PUT /api/image-annotation/template-library
```

Define one shared normalization and serialization path.

A template created or updated in either screen must be readable in the other.

Do not create a Diagram 2 template table, endpoint, local-storage copy, or second schema.

Renderer-specific preview thumbnails may be cached separately but must not be persisted as the canonical template.

## 6. Shared document adapter

Both screens must use the same existing Diagram backing documents.

Create a small shared adapter only if it reduces duplication and does not destabilize Diagram 1.

Responsibilities may include:

```text
List Diagram documents
Read Diagram image/state
Create backing document
Save backing document
Save Diagram info
Move document
Delete document
```

Do not duplicate database records for Diagram 2.

## Acceptance criteria

- Shared contracts are documented in code and tests.
- Diagram 1 still behaves the same.
- Existing exported Diagram files still import.
- Existing templates still load.
- Clipboard package can serialize, parse, validate, and remap IDs in pure tests.
- No Diagram 2 renderer exists yet.
