Before starting: Replace every value in angle brackets, confirm the maintenance window, and have a second person verify the backup paths and migration order. A database backup path is on the SQL Server machine and must be writable by the SQL Server service account.

1. Prepare the release and rollback copy

  1. Publish the current .NET application into a new, empty release folder. Do not publish directly over the live PMT folder.
  2. Copy the entire current live application folder to a dated rollback folder.
  3. Record the BDO values for the IIS site/application, application pool, SQL Server, live application folder, and UploadStorage.RootPath.
  4. Confirm the release contains all five SQL files listed in Step 5 and this runbook.
$ReleaseRoot   = '<FULL PATH TO THE NEW PUBLISHED PMT RELEASE>'
$LiveRoot      = '<FULL PATH TO THE LIVE PMT APPLICATION>'
$RollbackRoot  = '<FULL PATH TO THE DATED APPLICATION BACKUP>'
$MigrationRoot = Join-Path $ReleaseRoot 'Sql\Migrations'
$SqlServer     = '<SQL SERVER OR SQL SERVER\INSTANCE>'
$UploadRoot    = '<VALUE OF UploadStorage.RootPath>'
$UploadBackup  = '<FULL PATH TO THE DATED UPLOAD BACKUP>'
  • The new application has been published successfully.
  • The live application folder has been copied to the rollback folder.
  • The BDO appsettings.json, connection string, path base, and upload settings have been saved.

2. Stop PMT and back up user-uploaded files

  1. Put PMT into the agreed maintenance window.
  2. Stop the PMT IIS application pool in IIS Manager. Confirm the PMT site no longer serves requests.
  3. Copy the complete upload-storage folder to the dated upload backup. Preserve subfolders and file timestamps.
# Run after PMT is stopped. The destination must not be inside the source.
Copy-Item -LiteralPath $UploadRoot -Destination $UploadBackup -Recurse

# Compare the file counts. They must match.
(Get-ChildItem -LiteralPath $UploadRoot -File -Recurse).Count
(Get-ChildItem -LiteralPath $UploadBackup -File -Recurse).Count

3. Check and back up the PMT database

Run these statements in SQL Server Management Studio against the BDO SQL Server. Change only the backup path.

USE [master];
GO

DBCC CHECKDB ([PMT]) WITH NO_INFOMSGS;
GO

BACKUP DATABASE [PMT]
TO DISK = N'<SQL SERVER LOCAL BACKUP PATH>\PMT_Before_1.8_20260714.bak'
WITH COPY_ONLY, INIT, CHECKSUM, STATS = 10;
GO

RESTORE VERIFYONLY
FROM DISK = N'<SQL SERVER LOCAL BACKUP PATH>\PMT_Before_1.8_20260714.bak'
WITH CHECKSUM;
GO
Do not continue unless DBCC CHECKDB reports no errors, the backup completes, and RESTORE VERIFYONLY succeeds.

4. Confirm the exact migration files

Use the migration files from the same release as the published application:

  1. PMT_1.3_to_1.4.sql — internal invitation URLs
  2. PMT_1.4_to_1.5.sql — expanded user-role values
  3. PMT_1.5_to_1.6.sql — managed roles and stable role codes
  4. PMT_1.6_to_1.7.sql — role security, explicit user permissions, and admin password resets
  5. PMT_1.7_to_1.8.sql — safe attachment deletion
$Expected = @(
  'PMT_1.3_to_1.4.sql',
  'PMT_1.4_to_1.5.sql',
  'PMT_1.5_to_1.6.sql',
  'PMT_1.6_to_1.7.sql',
  'PMT_1.7_to_1.8.sql'
)

$Expected | ForEach-Object {
  $Path = Join-Path $MigrationRoot $_
  if (-not (Test-Path -LiteralPath $Path)) { throw "Missing migration: $Path" }
  Get-FileHash -Algorithm SHA256 -LiteralPath $Path
}

5. Apply the migrations in order

Run the following from an elevated PowerShell window on a machine that has sqlcmd and access to the BDO SQL Server. The -b option stops on SQL errors and -I enables the required quoted-identifier behavior.

$ErrorActionPreference = 'Stop'

foreach ($Name in $Expected) {
  $Path = Join-Path $MigrationRoot $Name
  Write-Host "Applying $Name"
  & sqlcmd -S $SqlServer -d master -E -b -I -i $Path
  if ($LASTEXITCODE -ne 0) {
    throw "Migration failed: $Name. Leave PMT stopped and follow the rollback section."
  }
}

Write-Host 'All five migrations completed.' -ForegroundColor Green

If BDO requires SQL authentication, replace -E with the approved secure authentication method. Do not place a database password in this HTML file, a command-history file, or source control.

If any script returns an error, stop. Do not skip the failed script and do not start the new application. Keep the site offline while you investigate or restore the verified backup.

6. Verify the migrated database before deploying the application

Rerun the data-manifest query from Step 3. The existing row counts must match the saved pre-migration results. New migration-owned tables will contain new configuration rows.

USE [PMT];
GO
SET NOCOUNT ON;

IF OBJECT_ID(N'[pmt].[UserInvitations]', N'U') IS NULL
  THROW 51101, 'Invitation migration is missing.', 1;
IF COL_LENGTH(N'pmt.Lookups', N'Code') IS NULL
  THROW 51102, 'Managed-role migration is missing.', 1;
IF OBJECT_ID(N'[pmt].[SecurityResources]', N'U') IS NULL
  THROW 51103, 'Security migration is missing.', 1;
IF OBJECT_ID(N'[pmt].[DeleteTaskAttachment]', N'P') IS NULL
  THROW 51104, 'Attachment-deletion migration is missing.', 1;
IF OBJECT_ID(N'[pmt].[AdminResetUserPassword]', N'P') IS NULL
  THROW 51105, 'Admin password-reset migration is missing.', 1;

IF EXISTS (
  SELECT 1
  FROM sys.sql_modules
  WHERE object_id IN (OBJECT_ID(N'[pmt].[UpsertLookup]'), OBJECT_ID(N'[pmt].[DeleteLookup]'))
    AND uses_quoted_identifier = 0
)
  THROW 51106, 'Managed-role procedure session settings are invalid.', 1;

SELECT N'Roles' AS [Configuration], COUNT(*) AS [RowCount]
FROM [pmt].[Lookups] WHERE [LookupType] = N'Role'
UNION ALL SELECT N'SecurityResources', COUNT(*) FROM [pmt].[SecurityResources]
UNION ALL SELECT N'RolePermissions', COUNT(*) FROM [pmt].[RolePermissions];

DBCC CHECKDB ([PMT]) WITH NO_INFOMSGS;
GO

Expected configuration counts for this release are 9 Roles, 14 Security Resources, and 112 Role Permissions. Existing BDO business-data counts must match the pre-migration manifest.

7. Deploy the published .NET application

  1. Keep the IIS application pool stopped.
  2. Copy the new published files into the live application folder, or switch IIS to the prepared release folder according to BDO's normal deployment practice.
  3. Restore BDO's saved production configuration values. In particular, preserve the production connection string, Deployment.PathBase, and all UploadStorage settings.
  4. Do not delete or replace the production upload-storage folder.
  5. Verify the IIS identity still has the required database and upload-folder access.
  6. Start the PMT IIS application pool and open PMT.
This release includes .NET/backend and database changes. It must be compiled and published; a browser refresh alone is not enough. After deployment, press Ctrl+F5 once in the browser so cached JavaScript and CSS cannot mask the new release.

8. Production smoke test

Use an administrator account and one non-administrator test account. Do not edit or delete real BDO work items just to perform a test.

  • Sign in and confirm Dashboard, Projects, Sprints, Dev Tasks, Bug Tracking, Scrum, Documentation, Kanban, Gantt, and Road Map load.
  • Open representative existing projects, sprints, tasks, bugs, Scrum entries, and Documentation entries; confirm their prior content is intact.
  • Open Settings → Roles; confirm all nine roles display. Create a temporary role, rename it, and delete it.
  • Assign a temporary role to the non-administrator test user and confirm PMT blocks deletion while that role is assigned. Restore the original role, then delete the temporary role.
  • Open Settings → Security; confirm resources, role permissions, and user overrides display. Verify an explicit No Access test override hides/blocks the selected resource, then remove the override.
  • From Settings → Users, use the administrator key action to reset the test user's password, then confirm the test user can sign in.
  • Generate an internal invite URL for one test project, open it in a private browser window, create a test profile with an avatar, and confirm the new user lands on the expected project or sprint screen.
  • On a temporary test task or Documentation entry, upload an attachment, delete it with the hover ×, confirm the prompt, save, reopen, and confirm it is gone.
  • Open representative read-only dialogs and confirm fields with no value are hidden.
  • Review the application and Windows event logs for unexpected errors.

Rollback plan

Use this only if the post-migration verification or production smoke test fails and the problem cannot be corrected safely inside the maintenance window.

  1. Stop the PMT IIS application pool.
  2. Preserve the failed deployment's logs for diagnosis.
  3. Restore the prior live application folder from $RollbackRoot.
  4. Restore the upload folder from $UploadBackup if any files changed after deployment began.
  5. Restore the verified Version 1.3 database backup with the SQL below.
  6. Restore the saved BDO production configuration and start the prior PMT application.
  7. Sign in and repeat the basic read-only smoke checks.
USE [master];
GO
ALTER DATABASE [PMT] SET SINGLE_USER WITH ROLLBACK IMMEDIATE;
GO
RESTORE DATABASE [PMT]
FROM DISK = N'<SQL SERVER LOCAL BACKUP PATH>\PMT_Before_1.8_20260714.bak'
WITH REPLACE, CHECKSUM, STATS = 10;
GO
ALTER DATABASE [PMT] SET MULTI_USER;
GO
DBCC CHECKDB ([PMT]) WITH NO_INFOMSGS;
GO
There are intentionally no downgrade scripts. A rollback restores the matched application folder, database backup, and uploaded-file backup as one set.
