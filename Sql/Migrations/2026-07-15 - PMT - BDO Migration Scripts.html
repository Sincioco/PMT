2. Confirm BDO's starting database version

Run this read-only query in SSMS before the maintenance window. A historical BDO Version 1.10 database can legitimately have no PMT_DatabaseVersion property. The required Version 1.10 marker is PMT_SecurityRoleDefaultsVersion = 1.10.

USE [PMT];
GO
SET NOCOUNT ON;

DECLARE @DatabaseVersion NVARCHAR(20) =
(
    SELECT CONVERT(NVARCHAR(20), [value])
    FROM sys.extended_properties
    WHERE [class] = 0 AND [name] = N'PMT_DatabaseVersion'
);

DECLARE @SecurityDefaultsVersion NVARCHAR(20) =
(
    SELECT CONVERT(NVARCHAR(20), [value])
    FROM sys.extended_properties
    WHERE [class] = 0 AND [name] = N'PMT_SecurityRoleDefaultsVersion'
);

SELECT
    COALESCE(@DatabaseVersion, N'<not recorded; valid for historical 1.10>') AS [PMT_DatabaseVersion],
    @SecurityDefaultsVersion AS [PMT_SecurityRoleDefaultsVersion],
    CASE
        WHEN @SecurityDefaultsVersion <> N'1.10' OR @SecurityDefaultsVersion IS NULL THEN N'STOP'
        WHEN @DatabaseVersion IS NULL OR @DatabaseVersion = N'1.10' THEN N'START WITH 1.10 TO 1.11'
        WHEN @DatabaseVersion IN (N'1.11', N'1.12', N'1.13') THEN N'STOP: INVESTIGATE PARTIAL DEPLOYMENT'
        WHEN @DatabaseVersion = N'1.14' THEN N'VERIFY PRIOR RELEASE; THEN START WITH 1.14 TO 1.15'
        WHEN @DatabaseVersion = N'1.15' THEN N'ALREADY CURRENT; DO NOT RERUN'
        ELSE N'STOP: UNKNOWN DATABASE VERSION'
    END AS [Decision];
GO

5. Save the pre-migration data manifest

Run the following in SSMS, save both result sets as dated CSV files, and keep them with the deployment record. The first query captures every PMT table. The second captures representative identifiers and timestamps without exporting private bodies.

USE [PMT];
GO
SET NOCOUNT ON;

SELECT
    [Schema].[name] AS [SchemaName],
    [Table].[name] AS [TableName],
    SUM([Partition].[row_count]) AS [RowCount]
FROM sys.tables AS [Table]
INNER JOIN sys.schemas AS [Schema]
    ON [Schema].[schema_id] = [Table].[schema_id]
INNER JOIN sys.dm_db_partition_stats AS [Partition]
    ON [Partition].[object_id] = [Table].[object_id]
WHERE [Schema].[name] = N'pmt'
  AND [Partition].[index_id] IN (0, 1)
GROUP BY [Schema].[name], [Table].[name]
ORDER BY [Schema].[name], [Table].[name];

SELECT N'Users' AS [Entity], [UserId] AS [Id], [Nickname] AS [Reference], [CreatedAt], [UpdatedAt]
FROM [pmt].[Users]
UNION ALL
SELECT N'Projects', [ProjectId], [Code], [CreatedAt], [UpdatedAt] FROM [pmt].[Projects]
UNION ALL
SELECT N'Sprints', [SprintId], [Code], [CreatedAt], [UpdatedAt] FROM [pmt].[Sprints]
UNION ALL
SELECT N'WorkTasks', [TaskId], [Code], [CreatedAt], [UpdatedAt] FROM [pmt].[WorkTasks];

SELECT N'SecurityResources' AS [Configuration], COUNT_BIG(*) AS [RowCount]

SELECT N'RolePermissions', COUNT_BIG(*) FROM [pmt].[RolePermissions];

SELECT

SELECT COUNT_BIG(*) AS [ExistingAbout3dSeedTitleCount]

INNER JOIN [pmt].[Projects] AS [Project] ON [Project].[ProjectId] = [Blog].[ProjectId]

foreach ($FileName in $ExpectedHashes.Keys) {
    $Path = Join-Path $MigrationRoot $FileName
    if (-not (Test-Path -LiteralPath $Path)) { throw "Missing migration: $Path" }

    $Text = [IO.File]::ReadAllText($Path) -replace "`r`n", "`n" -replace "`r", "`n"
    $Bytes = [Text.UTF8Encoding]::new($false).GetBytes($Text)
    $Sha256 = [Security.Cryptography.SHA256]::Create()
    try {
        $ActualHash = ([BitConverter]::ToString($Sha256.ComputeHash($Bytes))).Replace('-', '')
    }
    finally {
        $Sha256.Dispose()
    }

    if ($ActualHash -ne $ExpectedHashes[$FileName]) {
        throw "Hash mismatch for $FileName. Stop the deployment."
    }
}

7. Apply and verify each migration separately

Run this from an elevated PowerShell window on a machine with sqlcmd and access to BDO SQL Server. The runner reads the current version and begins with the immediately following migration, so a verified Version 1.14 installation runs only Version 1.15. Do not use that behavior to bypass the Step 2 investigation required for an unexpected partial deployment. The -b option returns a failure exit code for SQL errors; -I supplies the required quoted-identifier setting. A separate log is written for every step.

$ErrorActionPreference = 'Stop'
New-Item -ItemType Directory -Path $LogRoot -Force | Out-Null

$VersionQuery = "SET NOCOUNT ON; SELECT COALESCE((SELECT CONVERT(NVARCHAR(20), [value]) FROM sys.extended_properties WHERE [class]=0 AND [name]=N'PMT_DatabaseVersion'), N'1.10');"
$StartingVersion = (& sqlcmd -S $SqlServer -d $DatabaseName -E -b -I -h -1 -W -Q $VersionQuery).Trim()
if ($LASTEXITCODE -ne 0) { throw 'Could not read the starting PMT Database Version.' }

$AllSteps = @(
    @{ File = 'PMT_1.10_to_1.11.sql'; ExpectedVersion = '1.11' },
    @{ File = 'PMT_1.11_to_1.12.sql'; ExpectedVersion = '1.12' },
    @{ File = 'PMT_1.12_to_1.13.sql'; ExpectedVersion = '1.13' },
    @{ File = 'PMT_1.13_to_1.14.sql'; ExpectedVersion = '1.14' },
    @{ File = 'PMT_1.14_to_1.15.sql'; ExpectedVersion = '1.15' }
)

$StartIndex = switch ($StartingVersion) {
    '1.10' { 0 }
    '1.11' { 1 }
    '1.12' { 2 }
    '1.13' { 3 }
    '1.14' { 4 }
    '1.15' { throw 'PMT Database Version 1.15 is already current. Do not rerun the migration chain.' }
    default { throw "Unsupported PMT Database Version '$StartingVersion'. Stop the deployment." }
}

$Steps = @($AllSteps[$StartIndex..($AllSteps.Count - 1)])

foreach ($Step in $Steps) {
    $ScriptPath = Join-Path $MigrationRoot $Step.File
    $LogPath = Join-Path $LogRoot ($Step.File + '.log')

    Write-Host "Applying $($Step.File)"
    & sqlcmd -S $SqlServer -d $DatabaseName -E -b -I -i $ScriptPath -o $LogPath
    if ($LASTEXITCODE -ne 0) {
        throw "Migration failed: $($Step.File). Keep PMT stopped and inspect $LogPath."
    }

    $ActualVersion = (& sqlcmd -S $SqlServer -d $DatabaseName -E -b -I -h -1 -W -Q $VersionQuery).Trim()
    if ($LASTEXITCODE -ne 0 -or $ActualVersion -ne $Step.ExpectedVersion) {
        throw "Expected PMT Database Version $($Step.ExpectedVersion), found '$ActualVersion'."
    }

    Write-Host "Verified PMT Database Version $ActualVersion" -ForegroundColor Green
}

Write-Host 'PMT database migration 1.10 to 1.15 completed.' -ForegroundColor Green

If BDO requires SQL authentication, use its approved secure credential mechanism instead of -E. Never put a SQL password in this HTML, PowerShell history, a deployment log, or source control.

If any step fails, do not skip it, do not edit a version marker, and do not start either application version. The scripts are repair-and-rerun safe while their source or target version is current, but earlier scripts correctly reject a later database version. Restore the verified backup when a safe correction cannot be completed inside the maintenance window.

Rollback plan

Use rollback if database validation or the production smoke test fails and the issue cannot be corrected safely inside the maintenance window.

  1. Stop the PMT IIS application pool.
  2. Preserve the failed deployment's application, IIS, and SQLCMD logs.
  3. Restore the prior live application folder and saved production configuration from the matched rollback set.
  4. Restore the verified Version 1.10 database backup with the SQL below.
  5. Restore the upload backup only if files changed after the deployment started.
  6. Start the prior application and perform read-only login/navigation/data checks.
