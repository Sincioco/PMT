BDO Production: PMT Database Version 1.22 to 1.23

This one-time migration preserves BDO's current PMT Project as PMTQA, recreates only missing demo identities, restores the original SQL-seeded PMT Project for repeatable demonstrations, adds current-month vacation examples for shared PMT/LMS/HLS demo members, installs selected-date On Behalf Of attendance plus audited removal of explicit attendance entries, updates the post-deployment Development reset tools, installs a shared 13-template image-annotation catalog with optional per-user libraries that synchronize across browsers and computers, adds persistent manual hierarchy ordering for Diagram-backed Documentation, and seeds the pinned public PMT's Database Schema demo Diagram.

Never run Factory Reset PMT or SQL/03_SeedData.sql in Production. That path replaces shared data. The migration below is the only approved initial conversion.

Before running

  1. Schedule a maintenance window and stop the PMT application/IIS site so no writes occur.
  2. Take a full Production backup and run RESTORE VERIFYONLY against it.
  3. Confirm the database extended property PMT_DatabaseVersion is exactly 1.22.
  4. Confirm the active Project code is PMT and no Project already owns PMTQA.

Run the combined migration file

Open PowerShell in the repository's SQL\Migrations directory and run:

sqlcmd -S "<SQL SERVER OR SQL SERVER\INSTANCE>" -d PMT -E -b -I -i "PMT_1.22_to_1.23_All.sql"

The combined file includes the canonical PMT_1.22_to_1.23.sql step and stops on the first error. It creates [pmt].[ImageAnnotationDefaultTemplateLibraries] and seeds its single Version 1 library with 13 shared templates, creates [pmt].[GetImageAnnotationDefaultTemplateLibrary], and creates [pmt].[UserImageAnnotationTemplateLibraries] with [pmt].[GetUserImageAnnotationTemplateLibrary] and [pmt].[SaveUserImageAnnotationTemplateLibrary]. The user getter returns a saved personal override first, otherwise falls back to the shared catalog, and finally returns an empty Version 1 library only if neither row exists. Each versioned JSON library, including embedded image data, remains in SQL; the feature creates no upload-folder files for Maintenance to collect. The migration also adds [pmt].[Blogs].[SortOrder], returns it through [pmt].[GetAppState], and creates the owner-scoped [pmt].[MoveBlog] procedure. Existing and newly created Blog rows use the latest-first baseline order 0; manual moves assign explicit sibling order. The migration non-destructively inserts the public database-schema Diagram only when Sin does not already own an active Diagram with that title. Its authenticated Blog body embeds the SVG data URL rather than exposing the SQL metadata through a pre-login static URL. The SVG contains all 29 current [pmt] tables, 82 field-to-field foreign-key mappings, and one visible Entity Descriptions layer that users can hide from Objects. For SQL authentication, replace -E with the approved credential arguments. In SQL Server Management Studio, enable Query > SQLCMD Mode and run the same combined file.

The successful final message is:
PMT combined database migration 1.22 to 1.23 completed.

Future demo resets

  1. Open Settings > Development.
  2. Run Clear PMT Demo.
  3. Run Restore PMT Seed Data.

This cycle may be repeated. It resets only the demo Project identified by code PMT; PMTQA, BDO users, permissions, holidays, and private content remain protected.

Clear All Projects Except PMT is not part of the focused demo-reset cycle. It permanently deletes PMTQA and every other non-PMT Project. Private Documentation and private Logs survive Project cleanup only by being detached. Clear Users remaps private ownership to Sin, while Factory Reset PMT deletes all data, including private content and personal annotation-template libraries, before reseeding the shared annotation defaults and original demos.

Failure and rollback

The combined runner stops on the first SQL error, and the canonical migration uses one data transaction. Do not reopen PMT after a failure. Review the SQL error and confirm the database is still Version 1.22. A second execution after success is expected to stop safely because Version 1.23 is no longer a valid starting point. If the database state is uncertain, restore the verified pre-migration backup. After users begin changing PMT or PMTQA, do not attempt a reverse rename/delete rollback; use a forward correction.
