BDO Production: PMT Database Version 1.22 to 1.23

This one-time migration preserves BDO's current PMT Project as PMTQA, recreates only missing demo identities, restores the original SQL-seeded PMT Project for repeatable demonstrations, adds current-month vacation examples for shared PMT/LMS/HLS demo members, installs selected-date On Behalf Of attendance plus audited removal of explicit attendance entries, and updates the post-deployment Development reset tools.

Never run Factory Reset PMT or SQL/03_SeedData.sql in Production. That path replaces shared data. The migration below is the only approved initial conversion.

Before running

  1. Schedule a maintenance window and stop the PMT application/IIS site so no writes occur.
  2. Take a full Production backup and run RESTORE VERIFYONLY against it.
  3. Confirm the database extended property PMT_DatabaseVersion is exactly 1.22.
  4. Confirm the active Project code is PMT and no Project already owns PMTQA.

Run the combined migration file

Open PowerShell in the repository's SQL\Migrations directory and run:

sqlcmd -S "<SQL SERVER OR SQL SERVER\INSTANCE>" -d PMT -E -b -I -i "PMT_1.22_to_1.23_All.sql"

The combined file includes the canonical PMT_1.22_to_1.23.sql step and stops on the first error. For SQL authentication, replace -E with the approved credential arguments. In SQL Server Management Studio, enable Query > SQLCMD Mode and run the same combined file.

The successful final message is:
PMT combined database migration 1.22 to 1.23 completed.

Required verification

  1. Confirm the database version is 1.23.
  2. Confirm exactly one active PMTQA and one active PMT Project exist.
  3. Confirm the retained PMTQA Project has its original Project ID, 11 members, one Sprint, and 23 work items.
  4. Confirm the new PMT demo has six members, five Sprints, 31 work items, five Scrum rows, and 11 public Documentation rows.
  5. Confirm Bill, Sam, and Jensen each have an active vacation range in the current month. These people are shared PMT, LMS, and HLS demo members; the migration preserves any existing active overlapping range instead of adding a duplicate.
  6. Confirm PMTQA child codes begin with PMTQA- and PMT demo child codes begin with PMT-.
  7. Confirm the five recreated demo identities are active and do not use the public Password1 password. Administrators can impersonate them for role demonstrations.
  8. Confirm [pmt].[RecordAttendance] has the optional @AttendanceDate DATE parameter and [pmt].[RemoveAttendance] exists.
  9. Confirm [pmt].[DevelopmentClearProjectData] protects only Project code PMT, [pmt].[DevelopmentClearUsers] remaps audit actors and removes obsolete per-user permissions, and [pmt].[RequireDevelopmentSeedRestore] performs only the administrator check.
  10. In Scrum Calendar, confirm the seeded current-month vacations render for Bill, Sam, and Jensen, then record On Behalf Of attendance for a date other than today and confirm it appears on that calendar date. Confirm a self Check-In still uses the current UTC+8 workday.
  11. Remove an explicit Calendar attendance avatar and confirm only that row disappears and an Attendance / Removed audit event remains. Confirm removing a vacation avatar cancels its entire owner-only vacation range.
  12. Run DBCC CHECKCONSTRAINTS WITH ALL_CONSTRAINTS and DBCC CHECKDB (N'PMT') WITH NO_INFOMSGS.
  13. Start PMT and complete the targeted Projects, Dev Tasks, Bug Tracking, Scrum, Documentation, Settings Development, and Audit Trail smoke tests.

Future demo resets

  1. Open Settings > Development.
  2. Run Clear PMT Demo.
  3. Run Restore PMT Seed Data.

This cycle may be repeated. It resets only the demo Project identified by code PMT; PMTQA, BDO users, permissions, holidays, and private content remain protected.

Clear All Projects Except PMT is not part of the focused demo-reset cycle. It permanently deletes PMTQA and every other non-PMT Project. Private Documentation and private Logs survive Project cleanup only by being detached. Clear Users remaps private ownership to Sin, while Factory Reset PMT deletes all data, including private content, before reseeding the original demos.

Failure and rollback

The combined runner stops on the first SQL error, and the canonical migration uses one data transaction. Do not reopen PMT after a failure. Review the SQL error and confirm the database is still Version 1.22. A second execution after success is expected to stop safely because Version 1.23 is no longer a valid starting point. If the database state is uncertain, restore the verified pre-migration backup. After users begin changing PMT or PMTQA, do not attempt a reverse rename/delete rollback; use a forward correction.
